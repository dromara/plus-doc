# 国际化方案
- - -

## 前端国际化

参考 RuoYi 前端国际化文档：

- http://doc.ruoyi.vip/ruoyi-vue/document/htsc.html#前端国际化流程

补充说明：

- 当前前端项目已经内置 `zh_CN`、`en_US` 两套语言资源，目录位于 `plus-ui/src/lang`。
- 请求发送时会自动把当前语言写入请求头 `Content-Language`，后端无需前端每个接口手动重复传参。

## 后端国际化

- 2.7.0 以上支持后端国际化
- 3.4.0 以上支持 `Validator` 校验国际化

参考 demo 模块 `TestI18nController`。

请求头中增加 `content-language`，值需与国际化资源文件后缀一致，例如 `zh_CN`、`en_US`。

![输入图片说明](https://foruda.gitee.com/images/1678976722892396585/60917594_1766278.png "屏幕截图")

补充说明：

- 当前项目通过 `I18nLocaleResolver` 从请求头 `content-language` 解析语言环境。
- 如果请求头未传该值，则回退到服务端默认语言环境。
- 业务代码中可以直接使用 `MessageUtils.message("code")` 获取国际化文案。

![输入图片说明](https://foruda.gitee.com/images/1678976761015767874/729da3bc_1766278.png "屏幕截图")<br>

![输入图片说明](https://foruda.gitee.com/images/1678976765122587920/0b1027af_1766278.png "屏幕截图")<br>
![输入图片说明](https://foruda.gitee.com/images/1678976769965314387/0c416ede_1766278.png "屏幕截图")

补充说明：

- 校验消息使用 `{code}` 时，本质上还是走 Spring `MessageSource`，所以对应的资源文件里必须存在该 key。
- 自定义异常、业务提示、限流/幂等等公共组件的提示语，同样可以统一接入国际化资源，保持前后端文案一致。
